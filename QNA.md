## 🗄️ 1. 아키텍처 및 데이터베이스 (SQLite)

### Q1. 최신 라이브러리인 Room DB나 외부 서버 대신, 왜 안드로이드 순수 SQLite(`SQLiteOpenHelper`)를 사용하여 구축했나요?
* **답변**: 
  * 본 프로젝트의 주 목적은 **안드로이드 플랫폼이 제공하는 핵심적인 로컬 데이터베이스 제어 표준**을 이해하는 데 있습니다. 
  * Room DB는 내부적으로 코틀린 종속성을 강제하고 어노테이션 프로세서를 거쳐 빌드 설정을 복잡하게 만들 수 있습니다. 반면, 순수 `SQLiteOpenHelper`를 사용하면 추가 외부 라이브러리 설정 없이 오직 안드로이드 표준 SDK(Java)의 API만으로 동작하므로 빌드 리스크가 낮고, 쿼리가 실제로 데이터베이스 엔진에 전달되는 내부 메커니즘을 보다 명확히 학습하고 구현할 수 있어 이 방식을 채택했습니다.

### Q2. `SQLiteOpenHelper`에서 `onCreate()`와 `onUpgrade()`의 역할과 호출 시점은 언제인가요?
* **답변**:
  * **`onCreate()`**: 앱이 설치된 후 처음으로 데이터베이스에 접근을 시도할 때 단 한 번만 호출됩니다. 이 시점에 실질적인 테이블 생성 SQL 쿼리(`CREATE TABLE...`)가 동작합니다.
  * **`onUpgrade()`**: 데이터베이스 구조가 변경되어 생성자에 선언된 `DATABASE_VERSION` 상수가 기존 버전보다 높게 설정되었을 때 호출됩니다. 이 메서드 내에서 기존 테이블을 삭제(`DROP`)하거나 테이블 칼럼 구조를 변경(`ALTER TABLE`)하여 DB 스키마 버전 업데이트를 대응합니다.

### Q3. 데이터를 등록한 직후 메인 화면에서 목록 화면으로 전환하지 않고, 입력 필드를 비워둔 이유는 무엇인가요?
* **답변**:
  * **사용자 경험(UX) 관점**에서 최적화한 결과입니다. 사용자가 다수의 사이트 비밀번호를 연속해서 등록하고 싶을 때, 매번 목록 화면으로 강제 이동되거나 앱이 꺼지면 매번 다시 입력 화면을 켜야 하는 번거로움이 생깁니다. 
  * 따라서 저장 성공 즉시 입력 폼과 비밀번호 강도를 완전 초기화하고, 입력 포커스를 첫 번째 칸인 '사이트 이름'으로 강제 이동시켜 연속 가입 편의성을 극대화하고 다음 액션으로 사용자가 직접 "목록 보기"를 선택해 이동할 수 있게 설계했습니다.

## 🛡️ 2. 보안 기술 (Encryption & Security Flags)

### Q4. 비밀번호를 자동 생성할 때 왜 일반 `Random`이 아닌 `SecureRandom`을 사용했나요?
* **답변**:
  * 일반 `java.util.Random`은 단순한 산술 공식(선형 합동법) 기반으로 난수를 생성하여 속도는 빠르나, 출력된 몇 개의 난수 패턴을 알면 다음 생성될 난수를 수학적으로 쉽게 예측해낼 수 있어 암호학적으로 취약합니다.
  * 반면 **`java.security.SecureRandom`**은 예측 불가능성이 강하게 보장되는 **암호학적 의사난수 생성기(CSPRNG)**입니다. 운영체제(OS) 수준에서 엔트로피를 수집해 난수 시드를 만들기 때문에, 해커가 예측이나 역추적을 할 수 없어 비밀번호 생성 시스템에 필수적입니다.

### Q5. 추천받은 비밀번호가 무조건 '안전' 등급(4점 만점)을 획득하도록 보장한 로직은 어떻게 작성하셨나요?
* **답변**:
  * 기존의 난수 조립 방식은 아주 드문 확률로 특정 문자군(예: 특수문자나 숫자)이 누락되어 3점 이하의 등급이 나올 확률이 존재했습니다.
  * 이를 방지하기 위해 생성기 메서드 내부에 `while(true)` 루프 검증 코드를 추가했습니다. 난수 비밀번호를 생성한 직후, 저희가 정의한 **4가지 복잡도 기준(길이 8자 이상, 영문자 포함, 숫자 포함, 특수문자 포함)**을 자바 정규표현식(`matches()`)으로 확인하여 **모두 통과할 때만 루프를 탈출하여 화면에 반환**되도록 하였습니다. 이를 통해 100% 안전 등급의 패스워드 생성을 보장합니다.

### Q6. 캡처 방지 기능(`FLAG_SECURE`)은 어떻게 동작하며, 왜 에뮬레이터 개발 환경에서는 작동하지 않는 것처럼 보였나요?
* **답변**:
  * 액티비티 윈도우에 `FLAG_SECURE` 플래그를 등록하면, 안드로이드 OS가 해당 화면의 프레임 버퍼를 외부로 전송하는 것을 시스템 수준에서 원천 차단합니다. 따라서 기기 내부의 캡처 모듈이나 화면 녹화 앱이 동작할 수 없고, '최근 앱 보기' 화면에서도 내용이 검게 처리됩니다.
  * 다만, 에뮬레이터 자체의 우측 툴바 캡처 기능이나 PC 캡처 도구는 **안드로이드 OS 바깥의 영역(PC 윈도우 OS 그래픽 메모리)**에서 직접 가상 화면을 캡처하는 것이므로 차단 범위에 들어가지 않습니다. 실제 스마트폰 기기에서는 스크린샷 캡처가 완전히 불가능해집니다.

### Q7. 현재 비밀번호가 DB에 평문(Plain Text)으로 들어가는데, 실제 상용화한다면 이 보안 한계를 어떻게 해결할 수 있나요? (중요 질문)
* **답변**:
  * 현재는 SQLite DB의 학습 편의를 위해 평문으로 저장하였으나, 실제 상용 앱 수준으로 올린다면 기기 내부의 **Keystore(키스토어) 시스템**과 연동하여 **AES-256 대칭키 암호화 알고리즘**을 활용해 DB에는 암호화된 바이너리 텍스트 형태로 저장해야 합니다.
  * 사용자가 비밀번호 조회를 요청하여 화면에 띄울 때만 메모리 상에서 복호화해 보여주도록 아키텍처를 업그레이드할 것입니다.

## 🎨 3. UI 및 안드로이드 컴포넌트

### Q8. 리사이클러뷰(`RecyclerView`)에서 비밀번호 목록이 하나도 없을 때 빈 화면 처리는 어떻게 했나요?
* **답변**:
  * 레이아웃 상에서 리사이클러뷰 영역과 빈 상태 메시지 영역(`layout_empty_state`)을 겹쳐 구성(FrameLayout)해 두었습니다.
  * 데이터베이스에서 목록 데이터를 불러올 때 데이터의 개수가 `0`개라면, 리사이클러뷰의 Visibility(가시성)를 `GONE`으로 숨기고 빈 상태 뷰를 `VISIBLE`로 띄워 사용자가 앱의 현재 상태를 즉시 알 수 있게 구성했습니다.

### Q9. 목록에서 개별 아이템의 비밀번호 보이기/숨기기 토글은 어댑터에서 어떤 원리로 독립 구현되나요?
* **답변**:
  * 전체 목록을 다시 갱신하면 깜빡임 현상이 나고 성능 낭비가 큽니다. 따라서 `PasswordAdapter`의 `ViewHolder` 내부에 개별적으로 `boolean isPasswordVisible` 상태 플래그를 멤버 변수로 두어 개별 아이템 카드에서만 상태 변화가 일어나도록 독립시켰습니다.
  * 사용자가 눈 아이콘 클릭 시 해당 플래그를 토글하고, True이면 평문 텍스트를, False이면 도트(`••••`) 마스킹 스트링을 TextView에 실시간으로 매핑해 줍니다.
