## 🛡️ 1. 보안 기술 및 비즈니스 로직 (Security & Logic)

### Q1. 추천받은 비밀번호가 무조건 '안전' 등급(4점 만점)을 획득하도록 보장한 로직은 어떻게 작성하셨나요?
* **답변**:
  * 단순 난수 조립 시, 무작위 추출 특성상 특정 문자군(예: 특수문자나 숫자)이 누락되어 3점 이하의 취약한 등급이 나올 확률이 존재했습니다.
  * 이를 방지하기 위해 생성 메서드 내부에 `while(true)` 루프 검증 코드를 설계했습니다. 무작위 비밀번호를 생성한 직후, 저희가 정의한 **4가지 복잡도 기준(길이 8자 이상, 영문자 포함, 숫자 포함, 특수문자 포함)**을 자바 정규표현식(`matches()`)으로 확인하여 **모두 통과할 때만 루프를 탈출하여 화면에 반환**되도록 구현함으로써 100% 안전 등급의 패스워드 생성을 보장합니다.

### Q2. 캡처 방지 기능(`FLAG_SECURE`)은 어떻게 동작하며, 왜 에뮬레이터 개발 환경에서는 작동하는 것처럼 보였나요?
* **답변**:
  * 액티비티 윈도우에 `FLAG_SECURE` 플래그를 등록하면, 안드로이드 OS가 해당 화면의 그래픽 프레임 버퍼를 기기 외부로 유출하는 것을 시스템 수준에서 원천 차단합니다. 따라서 기기 내부의 캡처 숏컷이나 화면 녹화 앱이 동작할 수 없고, '최근 앱 보기' 화면에서도 화면이 안전하게 검게 가려집니다.
  * 다만, 에뮬레이터 자체의 우측 툴바 캡처 기능이나 PC 캡처 도구는 **안드로이드 OS 외부 영역(PC 윈도우 OS의 그래픽 메모리)**에서 직접 가상 머신의 화면을 긁어가는 것이므로 막을 수 없습니다. 실제 안드로이드 실기기에서는 스크린샷 캡처가 완벽히 차단됩니다.

### Q3. 현재 비밀번호가 DB에 평문(Plain Text)으로 들어가는데, 실제 상용화한다면 이 보안 한계를 어떻게 해결할 수 있나요? (핵심 질문)
* **답변**:
  * 현재는 데이터베이스 입출력 기본 원리 구현을 위해 평문으로 저장하였으나, 실제 상용 앱이라면 보안상 심각한 문제가 될 수 있습니다.
  * 실제 서비스 환경에서는 안드로이드 기기 내부의 안전한 하드웨어 격리 저장소인 **Keystore(키스토어) 시스템**과 연동하여 **AES-256 대칭키 암호화 알고리즘**을 구성하고, 비밀번호를 암호화된 형태로 DB에 저장한 뒤, 화면에 보여주는 필요한 시점에만 메모리상에서 일시적으로 복호화하는 2중 보안 구조를 적용할 것입니다.

## 🎨 2. UI 및 사용자 경험 (UI & UX)

### Q4. 데이터를 등록한 직후 메인 화면에서 입력 필드를 완전히 비우고 포커스를 첫 칸으로 돌린 기획 의도는 무엇인가요?
* **답변**:
  * **사용자 경험(UX) 최적화**를 위한 설계입니다. 사용자가 다수의 사이트 비밀번호를 연속적으로 편하게 등록할 수 있도록 하기 위함입니다.
  * 저장 성공 즉시 입력 폼과 비밀번호 강도를 완전 초기화하고, 입력 포커스를 첫 번째 칸인 '사이트 이름'으로 강제 이동시켜 터치 횟수를 최소화했습니다. 저장 후 사용자가 직접 다음 액션으로 "목록 보기" 버튼을 탭해 이동할 수 있도록 유연한 플로우를 제공합니다.

### Q5. 리사이클러뷰(`RecyclerView`)에서 비밀번호 목록이 하나도 없을 때 빈 화면 처리는 어떻게 구현했나요?
* **답변**:
  * 레이아웃 상에서 리사이클러뷰 영역과 빈 상태 안내 메시지 영역(`layout_empty_state`)을 겹쳐 구성(FrameLayout)해 두었습니다.
  * 데이터베이스에서 목록 데이터를 불러올 때 데이터의 개수가 `0`개라면, 리사이클러뷰의 가시성(Visibility)을 `GONE`으로 숨기고 빈 상태 안내 레이아웃을 `VISIBLE`로 띄워 사용자가 저장된 정보가 없음을 명확하게 인지하도록 처리했습니다.

### Q6. 목록에서 개별 아이템의 비밀번호 보이기/숨기기 토글은 어댑터에서 어떤 원리로 독립 구현되나요?
* **답변**:
  * 전체 목록을 다시 갱신하면 리스트 깜빡임 현상이 나고 성능 낭비가 발생합니다. 따라서 `PasswordAdapter`의 `ViewHolder` 내부에 개별적으로 `boolean isPasswordVisible` 상태 플래그를 멤버 변수로 두어 개별 아이템 카드에서만 상태 변화가 일어나도록 독립시켰습니다.
  * 사용자가 눈 아이콘 클릭 시 해당 뷰홀더의 플래그를 토글하고, True이면 평문 비밀번호를, False이면 도트(`••••`) 마스킹 스트링을 TextView에 매핑해 실시간 뷰 전환을 효율적으로 구현했습니다.
